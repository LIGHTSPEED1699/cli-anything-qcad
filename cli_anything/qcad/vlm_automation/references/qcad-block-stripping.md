# QCAD ODA Export BLOCK Stripping

## The Problem

QCAD Pro 3.32.x (and likely all versions using the ODA DWG translator) **silently strips the BLOCKS section** when exporting DXF → DWG. Title-block data, sheet metadata, ATTDEF defaults, and block attribute definitions are destroyed.

**Observed impact** (2026-05-13):
- Original DWG: **314,265 bytes**
- After QCAD DXF→DWG round-trip: **75,468 bytes** (~76 % data loss)
- BLOCKS-section line count identical in source and modified DXF, but absent in output DWG

## What Gets Lost

| Data type | Location in DXF | Preserved in QCAD DWG? |
|-----------|-----------------|------------------------|
| Modelspace entities (TEXT, LINE, INSERT, etc.) | ENTITIES | ✅ Yes |
| Layer definitions | TABLES/LAYER | ✅ Yes |
| Linetype definitions | TABLES/LINETYPE | ✅ Yes |
| Block definitions | BLOCKS | ❌ **No** |
| ATTDEF template text | Inside BLOCKS | ❌ **No** |
| ATTRIB values inside INSERTs | Inside BLOCKS | ❌ **No** |
| Title block revision rows | Inside `PLAINS-D-CAN` block | ❌ **No** |
| Sheet metadata | Inside block definitions | ❌ **No** |

## Why It Happens

The ODA DWG writer used by QCAD (`TD_Db.dll`) appears to discard block definitions that don't have corresponding INSERT references in modelspace, or it may restructure the block table in a way that loses ATTDEF/ATTRIB bindings. The `RDwgExporter` / `RDwgImporter` logs show no error for this — BLOCK data simply vanishes.

## Implications for the Pipeline

Any task requiring **editing BLOCK-level data** is **impossible** with the standard VLM-CAD QCAD headless pipeline:
- ❌ Adding/revising title block revision history rows
- ❌ Editing sheet numbers or drawing metadata stored in BLOCK attributes
- ❌ Modifying block definition geometry
- ❌ Preserving ATTDEF defaults for future user edits

Modelspace-only edits work fine:
- ✅ Deleting modelspace entities by handle
- ✅ Adding new modelspace entities (lines, text)
- ✅ Changing modelspace text content
- ✅ Layer visibility/color adjustments

## Decision Tree

```
Does the task require editing BLOCK data?
│
├─ YES → Do NOT use QCAD headless export
│        Options:
│        1. Edit original DWG directly with AutoCAD / full QCAD GUI
│        2. Use ODA File Converter with "preserve blocks" (unconfirmed)
│        3. Use LibreDWG dxf2dwg (if file originated from LibreDWG)
│        4. Accept BLOCK data loss; inform user explicitly
│
└─ NO  → QCAD headless pipeline is safe
           Use the standard tiered execution path
```

## Workarounds

### Workaround A: Modelspace Fallback
When the desired edit is a drawing number suffix (e.g. `00002-01` → `00002-02`), check whether the text exists in **modelspace** as well as inside a BLOCK.

Example (2026-05-13):
- BLOCK attribute: `022-122-97024-00002-01` inside `PLAINS-D-CAN` → destroyed on export
- Modelspace TEXT at handle `97B8`, x=18.73, y=10.11 with the same text → **editable via DXF**

Strategy: Edit the modelspace TEXT. AutoCAD users who explode the title block will still see the updated number. Title block attributes remain stale, but the visible modelspace copy is correct.

### Workaround B: LibreDWG dxf2dwg
If the original DWG was generated by LibreDWG, its DXF structure may round-trip through `dxf2dwg` with BLOCK data intact. However, LibreDWG's DXF output is often incompatible with AutoCAD 2018+ (see `references/libredwg-vs-oda-forensic-comparison.md`).

### Workaround C: ODA File Converter (Batch Mode)
The ODA File Converter AppImage can convert DXF → DWG with optional "preserve original drawing" settings. Requires `xvfb-run` for headless use. BLOCK preservation behavior is version-dependent — test on a backup first.

### Workaround D: Manual QCAD GUI
Open the modified DXF in QCAD Pro GUI, verify title block attributes are present (they should be — QCAD reads DXF BLOCKS correctly), then use **File → Export → DWG** with "Save block definitions" enabled. This is the only confirmed way to preserve BLOCK data, but requires interactive GUI access.

## Detection

After any DXF→DWG conversion, check file size:

```bash
orig_size=$(stat -c%s 'original.dwg')
new_size=$(stat -c%s 'output.dwg')
echo "Scale factor: $(echo "scale=2; $new_size / $orig_size" | bc)"
# If < 0.85, suspect BLOCK stripping or other structural loss
```

Also compare DXF section sizes before/after:

```python
import re

def count_section_lines(path, section_name):
    with open(path, 'r') as f:
        content = f.read()
    pattern = rf'SECTION\s+2\s+{section_name}.*?ENDSEC'
    match = re.search(pattern, content, re.DOTALL)
    if match:
        return match.group(0).count('\n')
    return 0

blocks_before = count_section_lines('modified.dxf', 'BLOCKS')
# If DWG output is missing ENTIRE BLOCKS section, QCAD stripped it
```

## In-Skill Mitigation

When a task targets BLOCK data, the skill should:
1. **Flag immediately** — "This edit touches BLOCK-level data. QCAD headless export will destroy it."
2. **Offer alternatives** — AutoCAD direct edit, QCAD GUI export, or accept the loss
3. **Never silently proceed** — the user will see a broken title block and assume the pipeline failed
