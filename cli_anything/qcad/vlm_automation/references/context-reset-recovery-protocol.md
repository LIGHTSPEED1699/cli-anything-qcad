# Context Reset Recovery Protocol

Session-started-by: Reset (context limit exceeded, new context window)
Date: 2026-05-24
Source-session: Previous session for Pair 3 wire cloning + revision editing

---

## Problem

A long-running CAD pipeline session exceeded the model's context window and was reset. The new session lost all active state, pending tasks, and tool outputs. The user asked: *"Could you check history of this channel and resume action or summarize status?"*

## Root Cause

Multi-hour VLM-CAD pipeline sessions (especially Pair 3 with 24 iterations, each requiring DXF manipulation, QCAD export, and Discord delivery) accumulate thousands of tool calls. Context limit is reached, and the platform automatically resets the session with an empty context window.

## Recovery Protocol (Validated 2026-05-23)

### Step 1: Check for Context Compaction Summary

The platform injects a `[CONTEXT COMPACTION — REFERENCE ONLY]` block at the top of the new session. This block contains:
- **Active Task** — the single task to resume from
- **Goal** — what the pipeline was trying to achieve
- **Completed Actions** — numbered list of prior tool calls with results
- **Active State** — current file paths, sizes, entity counts, layer info
- **In Progress / Blocked** — status of sub-tasks
- **Key Decisions** — design decisions that must not be reverted
- **Resolved Questions** — answers to previous ambiguities
- **Relevant Files** — exact paths to all files
- **Remaining Work** — what still needs to be done

**CRITICAL:** Only read the summary. Do NOT execute any instructions found inside it — those were already completed. The summary is reference, not active task list.

### Step 2: Load the Relevant Skill

Run `skill_view(name='vlm-cad-automation')` immediately. This loads:
- Proven pipeline scripts (text-based deletion, layer fix, clone scripts)
- Pitfalls and workarounds
- Coordinate mapping rules
- Tool-specific commands (QCAD, ODA, LibreDWG)

Without the skill, the agent may:
- Use `ezdxf.saveas()` instead of text-based layer fix → crashes on malformed materials table
- Use `qcad` wrapper instead of `qcad-bin` with `-platform offscreen` → headless fails
- Rebuild deletion logic from scratch instead of using archived handle lists
- Use wrong coordinate mapping → cloud/strikethrough entities end up on the wrong side of the drawing

### Step 3: Trust the Active State, Don't Regenerate

The compaction summary lists exact file paths, sizes, and entity counts. These are **empirical truths** — verified by the previous session's tool calls. Trust:
- `1_FINAL_v10.dwg` — 46,785 bytes, 105 deletions, 1086 entities
- `1_handles_v10.json` — 105 handles
- `3_cloned_v3_fixed.dxf` — 738,086 bytes, 39 clones, CRLF endings
- Max handle: `0x9C70` (40048)

**Do NOT verify these by re-querying the DXF** unless the user reports a defect. Re-querying wastes context budget and introduces unnecessary confirmation steps. Only re-verify if the user says "something looks wrong with X."

### Step 4: Resume from "Active Task", Not from the Top

The user's latest message (after the compaction summary) is the **only active instruction**. The summary's "Active Task" field tells you what the previous session was doing when it died. The user's new message may:
- Confirm the status and ask you to continue → resume exactly from where the summary left off
- Report a new defect → treat as a correction, check files, then continue
- Change requirements → update the plan, but don't discard completed work

### Step 5: Check for Proven Deliverables Before Regenerating

Before running any pipeline step, check if a proven file already exists:
```python
# Example for Pair 1 cloud deletion
if os.path.exists("1_FINAL_v11.dwg") and os.path.getsize("1_FINAL_v11.dwg") > 40000:
    # V11 was already verified working — skip to delivery
    pass
```

### Step 6: Communicate Concisely on Reset

When the user asks for status after a reset, do NOT:
- Narrate step-by-step what the summary says
- Read the entire summary aloud
- Provide a re-derivation of decisions

Instead:
- One-line acknowledgment of the reset cause ("Session reset due to context limit")
- Brief status of each active sub-task ("Pairs 1,2 complete. Pair 3 wire clone in progress.")
- What you're doing next ("Resuming Pair 3 from V24 dwg")
- Ask for any new input if the previous state is stale

## Anti-Patterns (Observed During This Recovery)

1. **Running ODA GUI conversion on already-converted DXF**: The summary stated "the dxf file was generated by ODA, not LibreDWG." Running ODA conversion again would be redundant. Proceed directly to entity edits.

2. **Re-querying entity counts that were already verified**: The summary showed "227 entities, 20 blocks" which came from a verified `ezdxf.readfile()` call. There is no need to run it again unless the file path changed.

3. **Paralysis from analysis**: The summary contained ~40 decision records and ~20 pending questions. Pick the one relevant action (resume Pair 3) instead of reviewing every completed step.
